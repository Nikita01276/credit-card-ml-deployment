# Архитектура сервиса
## Монолит или Микросервисы

Выбрана **монолитная архитектура** - весь код в одном Flask-приложении и одном Docker-контейнере

**Почему монолитная:**
- Сервис выполняет одну задачу - предсказание дефолта
- Проще в разработке и отладке

**Когда перейти на микросервисы:**
При масштабировании имеет смысл выделить отдельные сервисы: API Gateway, ML Inference, Feature Store, Logging, A/B Router.

---

##  RabbitMQ (концепт)

При батч-предсказаниях синхронный API не справится с нагрузкой. RabbitMQ решает это асинхронно:
```
Клиент, [очередь RabbitMQ], [Worker 1, 2...N], [База результатов]
```


##  Логирование

Каждый запрос логируется в JSON-формате:

```json
{
  "timestamp": "2026-01-01T12:00:00",
  "endpoint": "/predict",
  "request": {"LIMIT_BAL": 20000, "...": "..."},
  "response": {"prediction": 1, "probability": 0.77}
}
```

В production логи собираются через ELK-стек (Filebeat - Logstash - Elasticsearch - Kibana).

## ONNX (обзорно)

ONNX позволяет ускорить инференс и убрать зависимость от scikit-learn:

```python
from skl2onnx import convert_sklearn
onnx_model = convert_sklearn(model, initial_types=[...])
```

##  uWSGI + NGINX

Flask dev-сервер однопоточный — не подходит для production.

- **NGINX** — принимает соединения, балансирует нагрузку, завершает SSL
- **uWSGI** — запускает несколько воркеров Flask параллельно

---

##  DVC и MLflow (концепт)

- **DVC** - версионирование датасета. Файл хранится в облаке, в Git только указатель.
- **MLflow** - отслеживание экспериментов: параметры, метрики, артефакты моделей.
