# Архитектура сервиса

## 1. Монолит vs Микросервисы

### Выбранный подход: монолитная архитектура

В данном проекте выбрана **монолитная архитектура** — весь код (загрузка модели,
обработка запросов, логирование) упакован в одно Flask-приложение и один Docker-контейнер.

**Почему монолит подходит для этого проекта:**

- Сервис выполняет одну задачу — предсказание дефолта. Нет смысла дробить его на части.
- Проще в разработке и отладке: один процесс, одно логирование, один деплой.
- Достаточная производительность: Flask + joblib справляются с нагрузкой учебного проекта.
- Не требует оркестрации: нет зависимостей между отдельными сервисами.

**Когда стоило бы перейти на микросервисы:**

Если бы задача масштабировалась до production-системы крупного банка, имело бы смысл
выделить несколько независимых сервисов:

| Сервис | Ответственность |
|---|---|
| API Gateway | Приём запросов, аутентификация, маршрутизация |
| ML Inference Service | Загрузка модели и вычисление предсказаний |
| Feature Store | Обогащение входных данных историческими признаками |
| Logging Service | Сбор и хранение логов запросов |
| A/B Router | Распределение трафика между версиями модели |

Это позволило бы масштабировать только inference-сервис при росте нагрузки,
не затрагивая остальные компоненты.

## 2. Брокер сообщений (RabbitMQ — концепт)

В текущей реализации сервис работает **синхронно**: клиент отправляет запрос
и ждёт ответа. Это нормально для одиночных предсказаний в реальном времени.

**Проблема при масштабировании:**
Если банк захочет раз в день прогонять через модель миллион клиентов
(батч-предсказания), синхронный API с этим не справится.

**Решение с RabbitMQ:**

```
Клиент → [очередь задач в RabbitMQ] → [Worker 1]
                                     → [Worker 2]  → [База результатов]
                                     → [Worker N]
```

1. Клиент отправляет задание в очередь и сразу получает `task_id`.
2. Несколько воркеров параллельно забирают задания и делают предсказания.
3. Результат сохраняется в базе данных.
4. Клиент через `GET /results/{task_id}` забирает готовый результат.

**Дополнительные сценарии использования RabbitMQ:**
- Асинхронное логирование (не блокировать основной поток при записи логов)
- Уведомления: если `probability > 0.8` — отправить алерт риск-менеджеру

## 3. Логирование и мониторинг

### Текущая реализация

Каждый запрос к `/predict` и `/predict/ab` логируется в JSON-формате:

```json
{
  "timestamp": "2025-01-01T12:00:00+00:00",
  "endpoint": "/predict",
  "request": {"LIMIT_BAL": 20000, "SEX": 2, "...": "..."},
  "response": {"prediction": 1, "probability": 0.77, "model_version": "v1"}
}
```

### В production: ELK-стек

В промышленной системе логи собирались бы следующим образом:

```
Flask (JSON logs) → Filebeat → Logstash → Elasticsearch → Kibana
```

- **Filebeat** — агент, который читает лог-файлы и пересылает их дальше.
- **Logstash** — парсит и трансформирует логи.
- **Elasticsearch** — хранит и индексирует логи для быстрого поиска.
- **Kibana** — дашборд для визуализации: сколько запросов в минуту,
  какая доля предсказаний — дефолт, среднее время ответа и т.д.

## 4. ONNX-ML (обзорно)

**Зачем переводить модель в ONNX:**
ONNX (Open Neural Network Exchange) — универсальный формат для ML-моделей.
После конвертации модель работает быстрее (оптимизированный runtime)
и независима от Python и scikit-learn.

**Как конвертировать LogisticRegression:**
```python
from skl2onnx import convert_sklearn
from skl2onnx.common.data_types import FloatTensorType

initial_type = [('float_input', FloatTensorType([None, 23]))]
onnx_model = convert_sklearn(model, initial_types=initial_type)

with open("model_v1.onnx", "wb") as f:
    f.write(onnx_model.SerializeToString())
```

**Инференс через ONNX Runtime:**
```python
import onnxruntime as rt
sess = rt.InferenceSession("model_v1.onnx")
pred = sess.run(None, {"float_input": X})[0]
```

## 5. uWSGI + NGINX в production

**Проблема Flask dev-сервера:**
`app.run()` — это однопоточный сервер для разработки. В production он не справится
с параллельными запросами.

**Решение:**
```
Интернет → NGINX → uWSGI → Flask-приложение
```

- **NGINX** — принимает все входящие соединения, отдаёт статику сам,
  балансирует нагрузку между несколькими uWSGI-процессами, завершает SSL.
- **uWSGI** — запускает несколько воркеров Flask, обрабатывает запросы
  параллельно, управляет жизненным циклом процессов.

## 6. DVC и MLflow (концепт)

### DVC — контроль версий данных

В нашем проекте датасет `UCI_Credit_Card.csv` лежит в папке `data/`
и не добавляется в Git (файл слишком большой). DVC решает эту проблему:

```bash
dvc add data/UCI_Credit_Card.csv   # добавить файл под контроль DVC
dvc push                            # загрузить на удалённое хранилище (S3, GDrive)
```

Теперь в Git хранится только `data/UCI_Credit_Card.csv.dvc` (маленький файл-указатель),
а сам датасет — в облаке. Любой член команды делает `dvc pull` и получает нужную версию данных.

### MLflow — отслеживание экспериментов

При обучении нескольких моделей важно не потерять, какие гиперпараметры
дали какой результат. MLflow фиксирует это автоматически:

```python
import mlflow

with mlflow.start_run(run_name="RandomForest_v2"):
    mlflow.log_param("n_estimators", 100)
    mlflow.log_param("max_depth", 10)
    mlflow.log_metric("f1_score", 0.54)
    mlflow.sklearn.log_model(model, "model")
```

После этого в MLflow UI можно сравнить все эксперименты в виде таблицы
и выбрать лучшую модель для деплоя.

## 7. Бизнес-метрики

Помимо технических метрик (F1, Precision, Recall), банк оценивает модель
через призму финансового результата.

### Метрика 1: Ожидаемые предотвращённые потери (Expected Loss Prevented)

Банк теряет деньги, когда выдаёт кредит клиенту, который потом делает дефолт.
Если модель правильно выявляет таких клиентов — банк не выдаёт им кредит и избегает потерь.

```
ELP = Количество_TP × Средний_кредитный_лимит × Коэффициент_потерь
```

Где:
- **TP** — клиенты, которым модель правильно отказала (истинные дефолтники)
- **Средний кредитный лимит** ≈ 167 000 NT$ (по датасету)
- **Коэффициент потерь** ≈ 0.4 (банк теряет ~40% суммы при дефолте — типовое значение)

### Метрика 2: Доля одобренных заявок при том же уровне риска (Approval Rate)

Если модель точнее отсеивает реальных дефолтников (меньше False Positive),
банк может одобрить больше заявок от надёжных клиентов без увеличения риска.

```
Approval Rate = (Всего заявок - Отказы) / Всего заявок
```

Цель: при фиксированном допустимом уровне дефолтов (например, не более 5%)
максимизировать долю одобренных заявок.
