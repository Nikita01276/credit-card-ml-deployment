# План A/B-тестирования

## 1. Постановка задачи

**Цель:** определить, даёт ли модель v2 (Random Forest) статистически значимое
улучшение по сравнению с моделью v1 (Logistic Regression) в задаче
прогнозирования дефолта по кредитным картам.

| | Контрольная группа | Тестовая группа |
|---|---|---|
| **Модель** | v1 — Logistic Regression | v2 — Random Forest |
| **Группа** | control | treatment |
| **Доля трафика** | 50% | 50% |

---

## 2. Разделение трафика

Каждый входящий запрос к эндпоинту `POST /predict/ab` случайно направляется
на модель v1 или v2 с равной вероятностью 50%.

```python
version = random.choice(['v1', 'v2'])
```

Это **случайное** (не детерминированное) распределение — один и тот же клиент
при повторном запросе может попасть в другую группу.
В production-системе стоило бы закрепить группу за клиентом через хэш его ID:

```python
version = 'v1' if hash(client_id) % 2 == 0 else 'v2'
```

---

## 3. Продолжительность теста

| Параметр | Значение | Обоснование |
|---|---|---|
| Минимальная длительность | **2 недели** | Сглаживает недельную сезонность поведения клиентов |
| Минимальный размер выборки | **~1 000 запросов на группу** | Обеспечивает статистическую мощность теста ≥ 80% |
| Критерий остановки | Достижение размера выборки + прошло ≥ 2 недель | |

---

## 4. Метрики оценки

### Основная метрика: F1-score для класса дефолта (class=1)

F1-score — гармоническое среднее Precision и Recall. Выбрана как основная,
потому что учитывает оба типа ошибок:
- **False Positive** (выдали кредит дефолтнику) — финансовые потери банка
- **False Negative** (отказали надёжному клиенту) — упущенная прибыль

Формула: `F1 = 2 × (Precision × Recall) / (Precision + Recall)`

Текущие значения:
- v1 (Logistic Regression): **F1 = 0.46**
- v2 (Random Forest): **F1 = 0.54**

### Дополнительная метрика: Recall для класса дефолта

**Обоснование выбора Recall:**
В банковском кредитном скоринге пропустить реального дефолтника (False Negative)
обычно дороже, чем ошибочно отказать надёжному клиенту (False Positive).
Поэтому Recall важнее Precision: лучше перестраховаться, чем понести убытки.

Формула: `Recall = TP / (TP + FN)`

Текущие значения:
- v1: **Recall = 0.62**
- v2: **Recall = 0.56**

*Интересный момент: v1 имеет более высокий Recall, но более низкий F1.
Это означает, что v1 «осторожнее» (реже пропускает дефолтников),
но при этом чаще ошибочно отказывает надёжным клиентам.*

### Бизнес-метрика: Ожидаемые предотвращённые потери

```
ELP = TP_count × Avg_credit_limit × Loss_rate
```

Рассчитывается на основе выходов модели после окончания теста.

## 5. Статистический анализ

### Используемый тест: z-тест для разности двух пропорций

F1-score может быть представлен как пропорция при достаточно большой выборке.
z-тест применим, так как размер выборки > 30 и данные независимы между группами.

**Формулы:**

```
z = (p1 - p2) / sqrt(p * (1 - p) * (1/n1 + 1/n2))

где:
  p1, p2 — F1-score в группах v1 и v2
  p = (p1*n1 + p2*n2) / (n1 + n2) — объединённая оценка
  n1, n2 — размеры выборок
```

### Доверительные интервалы (95%)

```
CI = p ± 1.96 × sqrt(p * (1-p) / n)
```

Если доверительные интервалы для v1 и v2 не перекрываются —
разница статистически значима.

### Критерий успешности теста

Модель v2 считается **лучше**, если выполнены все три условия:

1. **F1-score v2 > F1-score v1** (основная метрика выше)
2. **p-value < 0.05** (разница статистически значима, не случайна)
3. **Относительный прирост F1 ≥ 5%** (практически значимое улучшение)

Если условия выполнены — переводим 100% трафика на модель v2.
Если нет — оставляем v1 и собираем больше данных.

## 6. Практическая демонстрация через API

### Запрос к A/B-эндпоинту
```bash
curl -X POST http://localhost:5000/predict/ab \
  -H "Content-Type: application/json" \
  -d '{
    "LIMIT_BAL": 50000, "SEX": 1, "EDUCATION": 2, "MARRIAGE": 2, "AGE": 35,
    "PAY_0": 0, "PAY_2": 0, "PAY_3": 0, "PAY_4": 0, "PAY_5": 0, "PAY_6": 0,
    "BILL_AMT1": 15000, "BILL_AMT2": 14000, "BILL_AMT3": 13000,
    "BILL_AMT4": 12000, "BILL_AMT5": 11000, "BILL_AMT6": 10000,
    "PAY_AMT1": 2000, "PAY_AMT2": 2000, "PAY_AMT3": 2000,
    "PAY_AMT4": 2000, "PAY_AMT5": 2000, "PAY_AMT6": 2000
  }'
```

### Возможные ответы

Если попал в контрольную группу (v1):
```json
{
  "prediction": 0,
  "probability": 0.23,
  "model_version": "v1",
  "model_type": "LogisticRegression",
  "ab_group": "control"
}
```

Если попал в тестовую группу (v2):
```json
{
  "prediction": 0,
  "probability": 0.18,
  "model_version": "v2",
  "model_type": "RandomForestClassifier",
  "ab_group": "treatment"
}
```

Поле `ab_group` позволяет при анализе разделить логи на две группы
и посчитать метрики по каждой.
